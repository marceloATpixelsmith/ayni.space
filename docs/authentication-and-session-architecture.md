# 05 — Authentication and Session Architecture

## Scope

- This document defines architecture constraints for its domain using `docs/monorepo-overview.md` as baseline and concrete repository paths as evidence.

## Confirmed

- Email/password signup now returns the same public success status/shape for both fresh and duplicate-email attempts (201 + generic next-steps message), while duplicate credential rows remain blocked internally and are only audit-classified server-side (`apps/api-server/src/routes/auth.ts`).
- Invitation create/resend APIs no longer return plaintext invitation tokens in JSON; raw tokens are now used only internally for lane1 email rendering/link generation (`apps/api-server/src/routes/invitations.ts`, `apps/api-server/src/lib/invitationEmail.ts`).
- Email/password signup UI now captures only `email` + `password` (no full-name and no confirm-password fields) while keeping strong-password validation aligned with backend policy (`apps/admin/src/pages/auth/Signup.tsx`, `lib/frontend-security/src/authValidation.ts`, `apps/api-server/src/lib/passwordAuth.ts`).
- Invitation acceptance no longer routes invitees through generic `/login` for password setup; invited users with `emailMode=create_password` now create password directly on invitation page with the same password policy and visibility-toggle input (`apps/admin/src/pages/auth/InvitationAccept.tsx`, `apps/api-server/src/routes/invitations.ts`).
- Post-auth onboarding now supports both organization onboarding (`/onboarding/organization`) and user onboarding (`/onboarding/user`) using app-context `requiredOnboarding` decisions (`apps/api-server/src/lib/appAccess.ts`, `apps/api-server/src/lib/postAuthRedirect.ts`, `apps/admin/src/App.tsx`, `apps/admin/src/pages/auth/Onboarding.tsx`).
- Login session persistence now supports an explicit `stayLoggedIn` flag (14-day maxAge) for both email/password and Google OAuth starts; default logins retain the standard idle timeout policy (`apps/admin/src/pages/auth/Login.tsx`, `lib/frontend-security/src/index.tsx`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/session.ts`).
- OAuth callback authorization now resolves active app context and normalized access profile before redirecting, then performs group-scoped denial cleanup on `access_denied` (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/appAccess.ts`).
- Auth entry surfaces (`/api/auth/google/url`, `/api/auth/login`, `/api/auth/forgot-password`) share the same app-context resolver + failure-code mapping, so fail-closed app-context outcomes return `400` with stable codes (for example `app_slug_missing`, `app_not_found`, `app_context_unavailable`) rather than `404` route-style misses (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/authContextPolicy.ts`).
- OAuth start/callback continuation now supports a sanitized root-relative `returnToPath` (for example invitation acceptance URLs), carries it inside signed OAuth state, and resolves with explicit precedence of MFA -> onboarding -> continuation -> default destination after successful callback session establishment (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/postAuthContinuation.ts`, `apps/api-server/src/lib/postAuthDestination.ts`, `apps/admin/src/pages/auth/Login.tsx`, `lib/frontend-security/src/index.tsx`).
- OAuth state now carries an encoded payload (`appSlug`, `returnTo`, `sessionGroup`, nonce) and callback validation fails closed when `appSlug` is missing or malformed, preventing callback authorization with incomplete app context (`apps/api-server/src/routes/auth.ts`).
- OAuth initiation accepts `intent` input, but callback redirect behavior is now explicitly intent-agnostic (no session `oauthIntent` plumbing).
- OAuth callback identity resolution now logs callback identity details, resolves users by `google_subject` first and email second, binds `google_subject` onto pre-provisioned rows when null, and denies unknown identities in `superadmin` access mode without auto-provisioning (`apps/api-server/src/routes/auth.ts`).
- Invitation acceptance UI now prevents duplicate turnstile-backed submissions for the same solved challenge and only resets the widget on turnstile-specific backend errors, preventing silent re-challenge loops (`apps/admin/src/pages/auth/InvitationAccept.tsx`, `lib/frontend-security/src/index.tsx`).
- Invitation acceptance route (`/invitations/:token/accept`) is intentionally reachable while unauthenticated: `ConfigDrivenAuthRoute` no longer pre-redirects invitation paths to `/login`, and invitation-specific next-step/login continuation handling is owned by shared `useInvitationAcceptRouteRuntime` composed by the `InvitationAccept` page (`apps/admin/src/App.tsx`, `apps/admin/src/pages/auth/InvitationAccept.tsx`, `lib/frontend-security/src/auth-page-orchestration.ts`).
- Invitation acceptance now starts Google OAuth directly from the invitation screen (with turnstile + `returnToPath` handoff), while still exposing password-login mediation via `/login?next=...`; password + MFA flows now persist invitation continuation through typed session continuation state (`pendingPostAuthContinuation` / `postAuthContinuation`) so invitation resumption survives MFA and onboarding (`apps/admin/src/pages/auth/InvitationAccept.tsx`, `apps/admin/src/pages/auth/Login.tsx`, `lib/frontend-security/src/index.tsx`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/postAuthContinuation.ts`, `apps/api-server/src/lib/session.ts`).
- Invitation acceptance/onboarding navigation is fail-closed when backend `nextPath` cannot be resolved; frontend invitation orchestration no longer injects root-path fallback redirects (`lib/frontend-security/src/auth-page-orchestration.ts`, `apps/admin/src/pages/auth/InvitationAccept.tsx`, `apps/api-server/src/routes/invitations.ts`).
- Hardcoded fallback destinations in auth routing are forbidden: auth route transitions must resolve via canonical post-auth destination logic and return explicit unresolved contracts (`POST_AUTH_DESTINATION_UNRESOLVED`) when destination resolution fails (`apps/api-server/src/lib/postAuthDestination.ts`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/routes/invitations.ts`, `lib/frontend-security/src/auth-page-orchestration.ts`).
- Invitation acceptance is now state-resolved before auth choice rendering: unauthenticated invitation pages call `GET /api/invitations/:token/resolve` to determine terminal invite status (`pending | expired | accepted | invalid`) and invited-email credential mode (`create_password` vs `sign_in`), and email-first invitees with no existing password can complete acceptance via `POST /api/invitations/:token/accept-email` (password setup + session establishment + membership activation) without first routing through generic login; resolution fetch/state machine/submission orchestration is centralized in `useInvitationAcceptRouteRuntime` (`apps/api-server/src/routes/invitations.ts`, `lib/frontend-security/src/auth-page-orchestration.ts`, `apps/admin/src/pages/auth/InvitationAccept.tsx`, `lib/frontend-security/src/index.tsx`).
- Invitation acceptance `emailMode=sign_in` now exposes an explicit email/password sign-in continuation link (`/login?next=/invitations/:token/accept`) alongside Google continuation, so existing invited users can complete the same invitation continuation through either credential lane (`apps/admin/src/pages/auth/InvitationAccept.tsx`, `lib/frontend-security/src/auth-page-orchestration.ts`).
- Invitation password acceptance now uses the same MFA gate contract as other first-auth success paths: `POST /api/invitations/:token/accept-email` returns MFA-pending responses (`mfaRequired`, `needsEnrollment`, `nextStep`) before any dashboard routing, and post-auth destination precedence is enforced as MFA first, then onboarding, then invitation continuation/default destination (`apps/api-server/src/routes/invitations.ts`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/postAuthFlow.ts`, `apps/api-server/src/lib/postAuthDestination.ts`, `lib/frontend-security/src/index.tsx`).
- Frontend post-auth navigation now fail-closes to an explicit allowlist (`/dashboard`/`/apps`, onboarding routes, MFA routes, invitation/event/client continuation routes) before any redirect from login/verify-email/MFA handlers, keeping frontend continuation handling aligned with backend continuation validation and preventing arbitrary in-app path injection from stale/malformed `nextPath` payloads (`lib/frontend-security/src/index.tsx`).
- Invitation resolution now resolves through `VITE_API_BASE_URL` when configured (same URL policy as shared API fetch), explicitly normalizes API-base values that already include `/api` to avoid `/api/api/...` resolve requests, validates resolve payload shape before rendering auth actions, and surfaces an explicit error state (without retry CTA) when resolve data is unavailable; this behavior is owned by shared route runtime orchestration and consumed by the admin invitation page (`lib/frontend-security/src/auth-page-orchestration.ts`, `apps/admin/src/pages/auth/InvitationAccept.tsx`).

- Backend authentication core is implemented in `apps/api-server/src/lib/auth.ts`.
- Authentication routes/session binding are implemented in `apps/api-server/src/routes/auth.ts`.
- Session lifecycle helpers are implemented in `apps/api-server/src/lib/session.ts` (canonicalized for store config, cookie options/clearing, session destruction, and logout-other-sessions cleanup).
- Session cookie policy is centralized in `apps/api-server/src/lib/session.ts` with production-safe cross-origin defaults (`SameSite=None`, `Secure=true` in production) and explicit trace logging (`[AUTH-CHECK-TRACE] COOKIE CONFIG ...`).
- Session-group resolution and group-specific cookie naming are centralized in `apps/api-server/src/lib/sessionGroup.ts` and consumed by auth routes in `apps/api-server/src/routes/auth.ts`.
- Session issuance is request-scoped by group: `apps/api-server/src/lib/session.ts` now resolves the group per request (`origin`/`referer` allowlist, OAuth callback `state`, or single matching group cookie) and dispatches to a per-group `express-session` middleware with a group-specific cookie name.
- Auth endpoints now also fail over to `appSlug`-derived group selection when trusted origin and session cookies are unavailable (`/api/auth/*` with `appSlug=admin` resolves `admin`, other app slugs resolve `default`), preventing admin MFA-pending logins from being created under the wrong cookie group during bootstrap edge cases (`apps/api-server/src/lib/sessionGroup.ts`).
- Session IDs are group-namespaced at issuance time (`<sessionGroup>.<uuid>` via `genid`) in `apps/api-server/src/lib/session.ts`, which prevents cross-group SID collisions in `platform.sessions` while still using one shared store table.
- Session group resolution fails closed for ambiguous multi-group cookie requests in sensitive flows (for example `/api/auth/logout`), preventing fallback to an arbitrary cookie when the request context is not trustworthy.
- Restricted app denial in OAuth callback now performs group-scoped cleanup: for the admin session group, non-super-admin users are redirected to `/login?error=access_denied` only after the admin-group session is destroyed and the admin-group cookie is cleared (`apps/api-server/src/routes/auth.ts`).
- Logout is group-scoped by default: `/api/auth/logout` resolves the current request's session group and only clears that group's cookie/session (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/session.ts`).
- OAuth callback state now encodes the session group prefix (`<group>.<uuid>`), which allows callback session selection to remain correct even if a browser already has multiple group cookies.
- Session persistence is a shared platform concern and is stored in `platform.sessions` (migration-managed), not `public.sessions`.
- Session persistence queries that revoke other sessions are group-isolated (`sessionGroup` match) rather than user-global, preventing cross-group revocation from `logout-others`.
- Frontend auth state, bootstrap, and route gating are implemented in `lib/frontend-security/src/index.tsx`.
- Frontend auth state machine now treats MFA-pending as distinct runtime states (`authenticated_mfa_pending_enrolled` and `authenticated_mfa_pending_unenrolled`) rather than collapsing pending sessions into fully authenticated or unauthenticated, and route guards now fail closed so protected app shells never render before MFA completion (`lib/frontend-security/src/index.tsx`, `apps/admin/src/App.tsx`).
- CSRF-aware shared fetch path is implemented in `lib/api-client-react/src/custom-fetch.ts`.
- OAuth start return-origin resolution accepts trusted `origin`/`referer` and trusted forwarded host/proto headers (`x-forwarded-host`, `x-forwarded-proto`) before allowlist validation so reverse-proxied admin logins derive correct app context at initiation (`apps/api-server/src/routes/auth.ts`).
- Frontend auth/session calls are hard-pinned to credentialed fetch mode in shared clients (`/api/auth/*` and `/api/csrf-token` always use `credentials: "include"` in `lib/api-client-react/src/custom-fetch.ts` and `lib/frontend-security/src/index.tsx`).
- CSRF handling is now transition-aware and shared: `@workspace/api-client-react` retries one time on `403 Invalid CSRF token` after invoking a frontend-registered CSRF refresher, and `AuthProvider` registers that refresher while deduplicating in-flight `/api/csrf-token` requests (`lib/api-client-react/src/custom-fetch.ts`, `lib/frontend-security/src/index.tsx`).
- Post-auth transition bootstrap is now explicit for password/verify-email/MFA continuation redirects: the frontend stamps an auth-transition marker before redirect and startup bootstrap treats that marker like OAuth grace, avoiding transient unauthenticated route decisions during session propagation (`lib/frontend-security/src/index.tsx`).
- MFA route guards now reuse shared authenticated next-step resolution (instead of hardcoded `/dashboard`) once a session is fully authenticated, preserving onboarding/access-denied routing decisions when users revisit `/mfa/*` after completion (`apps/admin/src/App.tsx`, `lib/frontend-security/src/index.tsx`).
- Backend app middleware composition runs through `apps/api-server/src/app.ts`, where session and security middleware ordering is centralized.
- Session middleware runtime provisioning is explicitly migration-managed (`createTableIfMissing=false`) and pinned to schema-qualified persistence (`platform.sessions`).

- OAuth callback app resolution is now strict: callback app context is derived from `state.appSlug` only, `getAppBySlug(appSlug)`/`getAppContext(userId, appSlug)` must succeed, and missing/invalid app slug state now returns explicit controlled login errors (`app_slug_invalid`, `app_not_found`, `app_context_unavailable`) instead of silent fallback redirects (`apps/api-server/src/routes/auth.ts`).
- Auth app-context resolution in `apps/api-server/src/lib/authContextPolicy.ts` now uses body/explicit/origin/session-group signals only to select a candidate slug; missing candidate context fails closed with `app_slug_missing`, and when a candidate exists canonical `getAppBySlug` lookup is required before policy derivation (`app_not_found` on real missing rows) with no slug-only synthesized policy or implicit admin fallback.
- Auth context input precedence is explicit and stable: request `appSlug` is highest priority, trusted origin/host-derived context is next, and session-group fallback is lowest; explicit appSlug remains authoritative over conflicting origin/session hints.
- Admin OAuth start is hardened at entry: admin policy + origin-derived context without explicit `appSlug=admin` is rejected fail-closed, while explicit valid admin `appSlug` succeeds (`apps/api-server/src/routes/auth.ts`).
- Organization-mode create-account callbacks now redirect to root-relative onboarding (`/onboarding/organization`) via `apps/api-server/src/lib/postAuthRedirect.ts` instead of legacy generic `/onboarding`.

- Auth UI shell/primitives are centralized in `lib/auth-ui/src/*` (AuthShell, divider, validation message, motion wrapper, Turnstile placement section, Google auth button), and admin auth pages consume that runtime layer directly (no app-local `apps/admin/src/pages/auth/components/*` auth shell wrappers remain). Shared `AuthShell` no longer hardcodes app-local image assets; branding is optional and caller-provided (`apps/admin/src/pages/auth/Login.tsx`, `apps/admin/src/pages/auth/Signup.tsx`, `apps/admin/src/pages/auth/InvitationAccept.tsx`, `lib/auth-ui/src/index.ts`, `lib/auth-ui/src/AuthShell.tsx`).
- Login/signup auth-route orchestration (post-auth redirect resolution, submit lifecycle + turnstile error handling, signup-affordance policy, metadata-driven signup-route redirect policy, signup verify-email redirect query assembly, and shared disabled-reason policy) is centralized in `lib/frontend-security/src/auth-page-orchestration.ts` via `useLoginRoutePolicy()`, `useLoginRouteActions()`, `useSignupRoutePolicy()`, `useSignupRouteActions()`, `getLoginDisabledReasons()`, and `getSignupDisabledReasons()`, leaving `apps/admin/src/pages/auth/Login.tsx` and `Signup.tsx` as thin route composition pages.
- Auth error/display and post-auth destination contracts are now normalized through shared/auth-layer constants instead of page-local string assumptions: backend/admin denied redirects use `ADMIN_ACCESS_DENIED_ERROR` codes, frontend pages resolve denied UI messaging via `@workspace/frontend-security` auth error helpers, and default post-auth fallback routing is centralized via shared `DEFAULT_POST_AUTH_PATH` constants in backend/frontend auth orchestration layers (`apps/api-server/src/lib/postAuthRedirect.ts`, `packages/auth/src/index.ts`, `apps/api-server/src/lib/postAuthRedirect.ts`, `apps/api-server/src/routes/auth.ts`, `lib/frontend-security/src/index.tsx`, `apps/admin/src/pages/auth/Login.tsx`).
- Frontend app-metadata resolution for auth route policy and signup affordances is now centralized in shared `useCurrentPlatformAppMetadata()` (`lib/frontend-security/src/index.tsx`) and consumed by admin app/auth route composition (`apps/admin/src/App.tsx`, `apps/admin/src/pages/auth/Login.tsx`, `apps/admin/src/pages/auth/Signup.tsx`) instead of app-local metadata-fetch effects.
- Auth boundary contracts are now consumed through shared helpers across backend/frontend/page layers: backend denied-login redirects are built through auth-layer constants (`buildAccessDeniedLoginPath`), frontend error query handling uses shared auth-code parsing (`parseAuthErrorCode`), login pages consume shared orchestration for denied cleanup/next-step/disabled-state (removing duplicate page-local policy glue), and signup route behavior delegates to shared auth-route policy/action contracts (`deriveAppAuthRoutePolicy`, shared submit gating/error handling, and verify-email redirect assembly) instead of superadmin-only/page-local fallbacks (`packages/auth/src/index.ts`, `apps/api-server/src/lib/postAuthRedirect.ts`, `apps/api-server/src/routes/auth.ts`, `lib/frontend-security/src/index.tsx`, `lib/frontend-security/src/auth-page-orchestration.ts`, `apps/admin/src/pages/auth/Login.tsx`, `apps/admin/src/pages/auth/Signup.tsx`, `apps/admin/src/pages/auth/accessDenied.ts`).
- Post-auth destination resolution is now explicitly backend-authoritative for access-denied branches: continuation paths are ignored unless app access is confirmed (`flowDecision.canAccess=true`), preventing stale invitation/onboarding continuations from overriding backend access denial outcomes (`apps/api-server/src/lib/postAuthDestination.ts`, `apps/api-server/src/routes/auth.ts`).
- Frontend authenticated next-step routing now mirrors backend denial precedence by applying superadmin/access-denied decisions before query-string continuation (`next`) paths, preventing stale continuations from masking deny outcomes in client-side guards (`lib/frontend-security/src/index.tsx`).
- Post-auth continuation restoration is now explicitly app-bound: continuation paths are resumed only when continuation `appSlug` matches resolved authenticated app context, and mismatched explicit continuation-type hints are rejected when path shapes do not match canonical continuation route patterns (`apps/api-server/src/lib/postAuthDestination.ts`, `apps/api-server/src/lib/postAuthContinuation.ts`, `apps/api-server/src/lib/postAuthFlow.ts`, `apps/api-server/src/routes/auth.ts`).
- Pending MFA session persistence is explicit in the shared session contract: `pendingUserId`, `pendingAppSlug`, `pendingMfaReason`, and `pendingStayLoggedIn` are persisted during MFA gate entry and consumed/cleared on completion (`apps/api-server/src/lib/session.ts`, `apps/api-server/src/routes/auth.ts`).
- Backend continuation validation is now fail-closed to a route allowlist (`/invitations/:token/accept`, event registration routes, client/public registration routes, and canonical app-entry routes) and rejects arbitrary paths; access-denied outcomes now always override continuation with denied-login routing, and password/verify-email MFA-pending responses include consistent `mfaRequired` + `needsEnrollment` + `nextStep` + `nextPath` fields (`apps/api-server/src/lib/postAuthContinuation.ts`, `apps/api-server/src/lib/postAuthFlow.ts`, `apps/api-server/src/lib/postAuthDestination.ts`, `apps/api-server/src/routes/auth.ts`).
- `/api/auth/me` now emits explicit machine-readable auth/session state markers for both pending and fully authenticated sessions (`authState`, `sessionState`, plus `mfaPendingReason` for pending MFA), and `requireAuth` does not update `lastSeenAt` on pending-MFA `/me` bootstrap reads (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/middlewares/requireAuth.ts`).
- CSRF/origin-referer middleware failures now expose stable machine-readable error codes (`CSRF_INVALID`, `ORIGIN_OR_REFERER_REQUIRED`, `ORIGIN_OR_REFERER_INVALID`), and frontend auth error mapping/CSRF retry detection consumes those codes instead of relying on fragile English error-text matching (`apps/api-server/src/middlewares/csrf.ts`, `lib/frontend-security/src/index.tsx`, `lib/api-client-react/src/custom-fetch.ts`).
- Signup Turnstile audit context no longer hardcodes `admin` as fallback app slug when request/session app context is absent; metadata now records null app context instead of implicit admin assumptions (`apps/api-server/src/middlewares/turnstile.ts`).
- Signup Turnstile audit context also no longer defaults unresolved session-group authority to `default`; when request/session group resolution is unavailable, metadata records `sessionGroup=null` to avoid hidden backend reassignment (`apps/api-server/src/middlewares/turnstile.ts`).

## Inferred

- Auth/session is designed as backend-authoritative with frontend providers/guards consuming backend session state.
- Session-group isolation is achieved end-to-end: cookie selection + cookie issuance + store revocation now all include group context, so unrelated groups are not globally logged out during denial/logout/revoke flows.
- Session rotation and active-org switching are tied to user/session flows (`apps/api-server/src/routes/users.ts` + `apps/api-server/src/lib/session.ts`).

## Unclear

- Full login method matrix and provider roadmap beyond current backend auth implementation.
- Whether any additional frontend shells (beyond admin) will share the same auth provider behavior.
- Whether future apps will require additional restricted session groups beyond `admin`.

## Intentional remaining gaps

- Session anomaly handling is currently observational (audit signal) and does not enforce adaptive/risk-based re-authentication.
- Rate limiting is currently in-process memory (production-safe defaults are in place, but no distributed/session-store-backed limiter yet).
- Turnstile remains intentionally targeted to public/high-risk auth entry points instead of blanket enforcement across every route.

## Do not break

- Do not move shared session storage out of `platform.sessions`; shared cross-app tables belong in `platform`, app-specific tables belong in app schemas.
- Do not bypass `lib/frontend-security` for auth gating in the admin frontend.
- Do not move auth/session logic outside the API server auth/session modules without updating route/middleware integration.
- Do not break CSRF/session coupling between `lib/frontend-security` and `lib/api-client-react/src/custom-fetch.ts`.
- Do not change middleware sequencing in `apps/api-server/src/app.ts` in ways that weaken auth/session enforcement.

## 2026-04-04 update — Email/password credential lane

### Confirmed

- Login session persistence now supports an explicit `stayLoggedIn` flag (14-day maxAge) for both email/password and Google OAuth starts; default logins retain the standard idle timeout policy (`apps/admin/src/pages/auth/Login.tsx`, `lib/frontend-security/src/index.tsx`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/session.ts`).
- Post-auth onboarding/access routing is auth-method agnostic: Google callback, password login, and post-MFA password completion all resolve their destination via one backend policy resolver (`apps/api-server/src/lib/postAuthFlow.ts`) that derives decisions from app context (`platform.apps` access profile + computed `requiredOnboarding`) and `getPostAuthRedirectPath` (`apps/api-server/src/lib/postAuthRedirect.ts`).
- Email/password is now additive to Google OAuth, with credentials and auth tokens split from `platform.users` into `platform.user_credentials` and `platform.auth_tokens` (`lib/db/src/schema/auth_credentials.ts`, `lib/db/migrations/20260404_email_password_auth.sql`).
- New auth endpoints were added for `signup`, `login`, `forgot-password`, `reset-password`, and `verify-email` in `apps/api-server/src/routes/auth.ts`.
- Password hashing now uses an explicit versioned `scrypt-v2$N=...,r=...,p=...$salt$digest` format for all new/updated credentials, and legacy temporary `scrypt$...` hashes are still verified and transparently upgraded to `scrypt-v2` on successful login (`apps/api-server/src/lib/passwordAuth.ts`, `apps/api-server/src/routes/auth.ts`).
- Password reset now revokes all other sessions for the user (cross-group) before clearing the current group cookie/session, reflecting account-recovery security scope rather than routine group-local logout scope (`apps/api-server/src/lib/session.ts`, `apps/api-server/src/routes/auth.ts`).
- Password auth rate limiting now applies both IP-keyed limits and opaque normalized-email keyed limits for signup/login/forgot-password without changing response shapes, so account enumeration posture is preserved while abuse resistance is stronger (`apps/api-server/src/middlewares/rateLimit.ts`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/passwordAuth.ts`).
- Auth/public sensitive endpoint throttling is now backed by shared Postgres state in `platform.rate_limits` when `NODE_ENV=production`, so rate-limit budgets are consistent across instances; non-production keeps in-memory buckets for local/dev ergonomics. Production continues to fail closed (`503 RATE_LIMIT_UNAVAILABLE`) for generic distributed-backend outages, but now falls back to an explicit emergency in-memory limiter (with actionable error logs) when rollout-specific DB errors indicate the `platform.rate_limits` relation/schema/privileges are missing (`apps/api-server/src/middlewares/rateLimit.ts`, `lib/db/migrations/20260407_distributed_rate_limits.sql`).
- Client network identity for abuse/rate-limit keying is derived from Express `req.ip` (which is controlled by explicit `trust proxy` policy) rather than directly consuming raw `x-forwarded-for` headers, preventing spoofable forwarded-header bypass when proxy trust is disabled (`apps/api-server/src/app.ts`, `apps/api-server/src/middlewares/rateLimit.ts`, `apps/api-server/src/lib/authAbuse.ts`).
- The case-insensitive users-email uniqueness migration now performs a deterministic preflight collision check and aborts with explicit remediation guidance when `lower(email)` duplicates exist, instead of failing unpredictably during index creation (`lib/db/migrations/20260404_users_email_case_insensitive_unique.sql`).
- Frontend auth UI now supports email/password in addition to Google OAuth (`apps/admin/src/pages/auth/*`) while preserving `lib/frontend-security` as the single auth state authority.
- Signup verification emails are now sent through the existing Lane 1 email pipeline (same provider/logging path as invitations) via `sendLane1AuthVerificationEmail` instead of direct auth-route sending (`apps/api-server/src/lib/invitationEmail.ts`, `apps/api-server/src/routes/auth.ts`).
- Signup verification and forgot-password reset emails now resolve via the shared Lane 1 email-template system (`platform.email_templates`) with app override + platform-default fallback, while preserving existing outbound log + provider send path (`apps/api-server/src/lib/emailTemplates.ts`, `apps/api-server/src/lib/invitationEmail.ts`, `apps/api-server/src/routes/auth.ts`).
- Email verification completion can now establish a password-auth session (or MFA pending session) when `appSlug` is provided, and resolves next-path via the same `resolvePostAuthFlowDecision` backend policy used by Google callback and password login (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/postAuthFlow.ts`).
- Verify-email frontend flow is now single-attempt per token/appSlug page load and fetches CSRF state before posting, while backend verify-token consumption differentiates invalid/expired/already-used outcomes so CSRF/system failures are not surfaced as token-expired errors (`apps/admin/src/pages/auth/VerifyEmail.tsx`, `lib/frontend-security/src/index.tsx`, `apps/api-server/src/routes/auth.ts`).

## 2026-04-04 update — Signup anti-abuse + MFA step-up architecture

### Confirmed

- Login session persistence now supports an explicit `stayLoggedIn` flag (14-day maxAge) for both email/password and Google OAuth starts; default logins retain the standard idle timeout policy (`apps/admin/src/pages/auth/Login.tsx`, `lib/frontend-security/src/index.tsx`, `apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/session.ts`).
- Signup now performs backend-only IPQS email risk scoring in `apps/api-server/src/lib/ipqs.ts` and applies decisioning (`allow` | `step_up` | `block`) inside `POST /api/auth/signup` (`apps/api-server/src/routes/auth.ts`).
- Email/password signup now emits structured audit decisions via `auth.signup.decision` with machine-readable reason codes (`disposable_email`, `undeliverable_email`, `ipqs_advisory_step_up`, `ipqs_provider_failure_step_up`, `turnstile_missing_or_invalid`, `duplicate_existing_email`, `signup_not_allowed_by_access_policy`, `validation_failed`, `internal_exception`) while keeping UI responses generic; signup Turnstile failures are also tagged with normalized reason codes in Turnstile audit metadata (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/middlewares/turnstile.ts`).
- IPQS provider errors/timeouts are fail-soft: signup continues as `step_up` (never blocked due to provider outage), and user is marked for MFA enrollment at first login.
- IPQS is advisory-only for non-definitive risk: high score, undeliverable, SMTP uncertainty, and other score-derived signals trigger `step_up` (email verification + MFA on first login) rather than hard-blocking signup.
- Hard-blocking based on IPQS is limited to definitive disposable-email detection; malformed/invalid input remains governed by existing validation logic.
- MFA is now first-class and TOTP-based (`apps/api-server/src/lib/mfa.ts`) with enrollment, challenge, and recovery-code flows under `/api/auth/mfa/*`.
- TOTP verification now decodes Base32 secrets with bit-buffer masking (preventing stale-bit corruption), applies identical verification logic in enrollment/challenge paths, and enforces a ±1 time-step acceptance window with replay-step rejection via `platform.used_mfa_totp_codes` (`apps/api-server/src/lib/mfa.ts`).
- Trusted devices are server-authoritative (`platform.trusted_devices`) with hashed token storage, secure cookie transport, and 20-day expiry.
- MFA issuer now derives from session-group authority (not app name/env): canonical display names are sourced from `platform.session_groups` and resolved by `apps/api-server/src/lib/sessionGroupDisplay.ts`.
- MFA issuer resolution now uses app-scoped runtime settings (`platform.app_settings` key `MFA_ISSUER`) as primary source with session-group display fallback, preserving existing behavior when an app-level issuer is absent (`apps/api-server/src/routes/auth.ts`, `apps/api-server/src/lib/runtimeSettings.ts`, `apps/api-server/src/lib/sessionGroupDisplay.ts`).
- Backend allowlist origins are derived from active `platform.apps.domain` values (protocol-normalized) and then merged with optional legacy `ALLOWED_ORIGINS` env entries; CORS/origin-referer checks consume that runtime layer (`apps/api-server/src/app.ts`, `apps/api-server/src/middlewares/csrf.ts`, `apps/api-server/src/lib/runtimeSettings.ts`).
- Required MFA enforcement now includes:
  - super admins (`platform.users.is_super_admin=true`),
  - org leadership roles mapped to real roles (`org_owner`, `org_admin`),
  - org client-registration pathways (organization apps with customer registration enabled).
