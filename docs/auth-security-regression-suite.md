- Access-mode auth route enforcement is now regression-protected across frontend route policy, auth-page rendering, backend onboarding resolution, and continuation replay:
  - `superadmin`
    - `/login` renders Google OAuth only.
    - Email/password login UI does not render.
    - Forgot-password affordances do not render.
    - Signup/create-account affordances do not render.
    - `/signup` redirects to `/login`.
    - `/forgot-password` redirects to `/login`.
    - `/reset-password` rejects non-superadmin credential restoration/session recovery.
    - `/verify-email` rejects non-superadmin account restoration/session issuance.
    - Non-superadmin users entering through OAuth callback, verify-email continuation, or authenticated route guards are redirected through explicit access-denied handling.
  - `organization`
    - Email/password login remains available.
    - Forgot-password remains available.
    - Signup/create-account visibility follows canonical `customerRegistrationEnabled` policy.
    - Invitation acceptance remains reachable while unauthenticated.
    - Invitation continuation survives login, MFA, onboarding, and verify-email transitions.
    - Organization onboarding is only reachable for legitimate org-registration flows.
  - `solo`
    - Email/password login remains available.
    - Forgot-password remains available.
    - Signup/create-account remains available.
    - Solo onboarding resolves only through `/onboarding/user`.
    - Solo users no longer fall through organization onboarding routing.

- Frontend/backend auth-route policy derivation alignment is now part of regression expectations:
  - `staffInvitesEnabled`
    - controls invitation-route affordances/policy visibility.
  - `customerRegistrationEnabled`
    - controls signup/create-account visibility and onboarding eligibility.
  - Frontend policy derivation must not hardcode organization invitations or customer-registration as universally enabled.

- Post-auth continuation regression coverage must preserve:
  - canonical precedence:
    - MFA
    - onboarding
    - continuation
    - default destination
  - app-bound continuation replay
  - fail-closed rejection of invalid continuation paths
  - allowlisted continuation-only paths:
    - `/invitations/:token/accept`
    - `/events/:id/register`
    - `/event-registration/:id/register`
    - `/register/client`
    - `/registration/client`
    - `/register/public`
    - `/registration/public`
    - `/`
    - `/dashboard`
    - `/dashboard/apps`
    - `/apps`
  - client/public registration continuation paths must not implicitly become generic signup or organization-onboarding flows unless dedicated registration implementation explicitly exists.

- Frontend runtime auth-route regression expectations now include:
  - superadmin login affordance hiding
  - signup/forgot-password route redirect enforcement
  - onboarding route separation between organization and solo apps
  - invitation continuation replay across login/MFA/onboarding
  - access-denied redirect correctness
  - app-mode-aware auth-page rendering
